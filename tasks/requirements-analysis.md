# 딱(Ddak) MVP 상세 요구사항 분석

## 1. 목적

이 문서는 `BRAINSTORM.md`를 구현 가능한 요구사항으로 구체화한다. 목표는 사용자가 큰 목표를 입력하면 AI가 2분 안에 시작할 수 있는 마이크로 태스크로 분해하고, 분류된 감정 태그에 따라 3D 또는 2D 오브젝트가 반응하는 MVP를 만드는 것이다.

## 2. 현재 구현 기준선

| 항목 | 현재 상태 | 구현 영향 |
| --- | --- | --- |
| 프레임워크 | Next.js 16.2.9 App Router | 브레인스토밍의 Next.js 14 표기는 현재 설치 버전 기준으로 보정 필요 |
| React | React 19.2.4 | 클라이언트 인터랙션 컴포넌트는 명시적으로 분리 |
| 스타일 | Tailwind CSS 4 | `globals.css`는 최소 설정만 있음 |
| 언어/검사 | TypeScript strict, ESLint | 타입 우선 설계 필요 |
| 패키지 매니저 | npm (`package-lock.json`) | 의존성 추가 시 npm 사용 |
| 화면 | `src/app/page.tsx`에 대시보드 플레이스홀더 | MVP 홈/태스크 플로우로 교체 필요 |
| 미설치 핵심 의존성 | OpenAI, Zustand, Framer Motion, Three.js, React Three Fiber, drei | 구현 시작 전 추가 필요 |

## 3. 제품 원칙

1. 사용자는 전체 계획이 아니라 "지금 딱 하나"만 보아야 한다.
2. AI 결과는 실천 가능한 물리적 행동이어야 하며, 각 행동은 2분 안에 시작 또는 완료 가능해야 한다.
3. 감정 태그는 태스크 세션당 단일 태그만 사용한다.
4. 3D 오브젝트는 장식이 아니라 감정 분류 결과를 보여주는 핵심 피드백이다.
5. 저성능, 모바일, 모션 민감 사용자에게는 2D/저모션 폴백을 제공한다.
6. 완료 검증은 명예 시스템으로 처리하고 별도 생산성 추적을 강제하지 않는다.

## 4. MVP 범위

### 4.1 포함

- 목표 입력 UI
- `/api/analyze` 서버 Route Handler
- OpenAI 기반 태스크 분해
- 단일 감정 태그 분류
- 5~10개 마이크로 태스크 생성
- 첫 번째 태스크 중심 표시
- 나머지 태스크 흐림 처리
- 2분 액션 강조
- 시작 버튼과 파티클/폭발 효과
- 완료 버튼
- 완료 후 기분 이모지 체크와 선택값 저장
- 완료 상태에 따른 오브젝트 변화
- 전체 완료 축하/해산 애니메이션
- LocalStorage 기반 미완료 세션 복귀
- 모바일/저성능 2D CSS 폴백
- 기본 오류, 빈 입력, API 제한 처리

### 4.2 제외

- 계정, 로그인, 서버 저장 동기화
- 히스토리 목록
- 소셜 공유
- 스트릭/습관 추적
- 실제 2분 타이머 강제
- 완료 여부 검증
- 다중 감정 태그 혼합 오브젝트
- 이모지 기반 다음 태스크 재작성 또는 추가 AI 호출
- 관리자/분석 대시보드

## 5. 핵심 사용자 플로우

### 5.1 신규 목표 시작

1. 사용자가 홈 화면에서 큰 목표를 입력한다.
2. 입력값이 비어 있으면 제출하지 않고 같은 화면에서 안내한다.
3. 제출 시 로딩 화면으로 전환한다.
4. 클라이언트는 `/api/analyze`에 목표를 전송한다.
5. 서버는 AI 응답을 검증 가능한 JSON으로 받아 태스크 목록과 감정 태그를 반환한다.
6. 클라이언트는 세션 상태를 저장하고 첫 번째 태스크 화면으로 전환한다.

### 5.2 지금 할 일 화면

1. 현재 태스크 카드에 2분 액션 하나를 크게 보여준다.
2. "2분이면 돼요" 같은 압박 완화 문구를 함께 보여준다.
3. 남은 태스크는 제목만 흐릿하게 보여주거나 개수만 보여준다.
4. 감정 태그에 맞는 3D 오브젝트 또는 2D 폴백이 배경에서 움직인다.
5. 사용자가 "시작!"을 누르면 시작 애니메이션을 실행하고 태스크 상태를 `started`로 바꾼다.

### 5.3 완료와 다음 단계

1. 사용자가 "완료했어요"를 누른다.
2. 기분 이모지 체크를 표시하되, 사용자는 선택하거나 건너뛸 수 있다.
3. 선택한 이모지는 현재 태스크 결과에 저장한다. 건너뛰면 `mood`는 비워 둔다. MVP에서는 이미 생성된 다음 태스크 문구를 이모지로 재작성하지 않는다.
4. 오브젝트는 조금 더 부드럽고 가벼운 상태로 변화한다.
5. 다음 태스크가 현재 태스크로 올라온다.
6. 모든 태스크가 완료되면 오브젝트 해산/폭발 애니메이션과 축하 메시지를 보여준다.

### 5.4 재방문 복귀

1. 페이지 진입 시 LocalStorage에 미완료 세션이 있는지 확인한다.
2. 미완료 세션이 있으면 "계속할까요?" 프롬프트를 보여준다.
3. 계속하기를 선택하면 마지막 `currentIndex`로 복귀한다.
4. 새로 시작하기를 선택하면 기존 세션을 명시적으로 폐기한다.

## 6. 기능 요구사항

| ID | 요구사항 | 수용 기준 |
| --- | --- | --- |
| FR-001 | 홈 화면에서 목표를 입력할 수 있다. | 입력창과 제출 버튼이 있고 모바일에서도 한 화면 안에 보인다. |
| FR-002 | 빈 입력이나 길이 제한 밖 입력은 분석 요청을 보내지 않는다. | 1~300자만 허용하고, 벗어나면 사용자에게 간단한 안내 문구를 보여준다. |
| FR-003 | 목표 제출 시 서버 API를 호출한다. | 클라이언트에 OpenAI 키가 포함되지 않는다. |
| FR-004 | AI는 5~10개 마이크로 태스크를 생성한다. | 응답 검증에서 범위를 벗어나면 오류로 처리한다. |
| FR-005 | 각 태스크는 단일 물리적 행동으로 표현된다. | "조사하기", "계획하기"처럼 모호한 표현보다 "파일 열기", "첫 문장 쓰기" 수준이어야 한다. |
| FR-006 | 각 태스크는 2분 안에 시작 또는 완료 가능해야 한다. | 프롬프트와 서버 검증 문구에 2분 제약을 포함한다. |
| FR-007 | AI는 단일 감정 태그를 반환한다. | `burden`, `creative`, `difficult`, `urgent`, `routine`, `new` 중 하나만 허용한다. |
| FR-008 | 첫 번째 태스크만 강조 표시한다. | 현재 태스크 카드가 시각적 중심이며 나머지는 흐림 처리된다. |
| FR-009 | 시작 버튼은 시각적 보상을 제공한다. | 버튼 클릭 후 파티클 또는 폭발 효과가 1회 재생된다. |
| FR-010 | 완료 버튼은 현재 태스크를 완료 처리한다. | 완료 후 `currentIndex`가 다음 미완료 태스크로 이동한다. |
| FR-011 | 완료 후 기분 이모지를 선택하거나 건너뛸 수 있다. | 선택값이 현재 태스크 결과에 저장된다. 건너뛰면 `mood`는 비워 두고 다음 태스크로 진행한다. MVP에서는 기록만 하고 다음 태스크 톤 조정은 후속 범위로 둔다. |
| FR-012 | 감정 태그는 오브젝트 프로필을 결정한다. | 태그별 색상, 형태, 움직임이 일관되게 매핑된다. |
| FR-013 | 완료 진행률은 오브젝트 변화에 반영된다. | 완료율이 올라갈수록 오브젝트가 더 부드럽고 가벼워진다. |
| FR-014 | 3D 사용이 어렵다면 2D CSS 폴백을 사용한다. | WebGL 미지원, 모바일 소형 화면, 낮은 메모리, `prefers-reduced-motion`에서 무리한 3D 렌더링을 피한다. |
| FR-015 | 미완료 상태는 LocalStorage에 저장된다. | 새로고침 후 마지막 진행 상태로 복구할 수 있다. |
| FR-016 | 새 목표 시작 시 기존 세션을 정리할 수 있다. | 사용자가 명시적으로 새 시작을 선택해야 기존 세션이 폐기된다. |
| FR-017 | API 호출은 rate limit 대상이다. | MVP는 단일 인스턴스 기준 인메모리 제한으로 IP당 분당 5회를 초과하면 429 응답을 반환한다. |
| FR-018 | API 오류는 사용자에게 복구 가능한 상태로 표시된다. | 재시도 버튼 또는 입력 화면 복귀가 가능하다. |
| FR-019 | 모든 주요 인터랙션은 키보드로 접근 가능해야 한다. | 입력, 제출, 시작, 완료, 이모지 선택이 포커스 가능하다. |
| FR-020 | 언어는 MVP에서 한국어를 기본으로 한다. | 주요 UI 문구와 AI 프롬프트 출력은 한국어 기준이다. |

## 7. 비기능 요구사항

| 영역 | 요구사항 |
| --- | --- |
| 보안 | OpenAI API 키는 서버 환경 변수로만 접근한다. 클라이언트 번들, LocalStorage, 로그에 키를 남기지 않는다. |
| 개인정보 | MVP는 계정 없이 LocalStorage에만 진행 상태를 저장한다. 민감한 목표 입력은 서버에 영구 저장하지 않는다. |
| 성능 | 초기 화면은 3D 번들에 막히지 않아야 한다. 3D 씬은 클라이언트 전용 지연 로딩을 검토한다. |
| 접근성 | 모션 민감 사용자를 위해 저모션 모드를 제공하고, 색만으로 상태를 구분하지 않는다. |
| 안정성 | AI 응답은 서버에서 스키마 검증 후 반환한다. 잘못된 응답을 성공처럼 처리하지 않는다. |
| 타입 안정성 | 감정 태그와 태스크 상태는 union type으로 제한한다. 임의 문자열과 `any` 사용을 피한다. |
| 반응형 | 모바일 우선으로 카드, 입력, 이모지 선택, 2D 폴백이 동작해야 한다. |

## 8. 데이터 모델 초안

```ts
type EmotionTag =
  | "burden"
  | "creative"
  | "difficult"
  | "urgent"
  | "routine"
  | "new";

type TaskStatus = "pending" | "started" | "done";

type MoodEmoji = "happy" | "frustrated" | "tired" | "neutral" | "proud";

type MicroTask = {
  id: string;
  title: string;
  twoMinuteAction: string;
  status: TaskStatus;
  mood?: MoodEmoji;
  completedAt?: string;
};

type TaskSession = {
  id: string;
  goal: string;
  emotionTag: EmotionTag;
  tasks: MicroTask[];
  currentIndex: number;
  createdAt: string;
  updatedAt: string;
  completedAt?: string;
};
```

## 9. API 계약 초안

### 9.1 `POST /api/analyze`

Request:

```json
{
  "goal": "세금 신고서 작성하기"
}
```

Success response:

```json
{
  "goal": "세금 신고서 작성하기",
  "emotionTag": "burden",
  "tasks": [
    {
      "title": "홈택스 열기",
      "twoMinuteAction": "지금 브라우저에서 국세청 홈택스 사이트만 열어봐."
    }
  ]
}
```

서버 응답의 `tasks`는 분석 결과 원문이다. 클라이언트는 검증된 응답을 세션으로 만들 때 `id`를 생성하고 `status: "pending"`으로 초기화한다. 서버에서 상태를 내려주지 않으면 태스크 진행 상태와 AI 분석 결과의 책임 경계가 명확해진다.

성공 응답의 `tasks`는 항상 5~10개여야 한다. 위 JSON은 항목 구조를 보여주는 예시이며, 실제 API는 응답 검증을 통과한 전체 태스크 배열만 반환한다.

Error responses:

| Status | 조건 | 응답 의미 |
| --- | --- | --- |
| 400 | goal 없음, 너무 짧음, 너무 김 | 클라이언트 입력 오류 |
| 429 | MVP 단일 인스턴스 기준 IP별 분당 5회 초과 | 잠시 후 재시도 필요 |
| 500 | AI 응답 실패 또는 스키마 불일치 | 서버 처리 실패 |

### 9.2 서버 프롬프트 제약

- 출력은 JSON만 허용한다.
- 감정 태그는 허용된 enum 중 하나만 반환한다.
- 태스크는 5~10개만 반환한다.
- 각 태스크는 사용자가 지금 바로 할 수 있는 하나의 물리적 행동이어야 한다.
- 각 태스크는 2분 안에 시작 또는 완료 가능해야 한다.
- 완벽한 완성보다 시작 마찰을 줄이는 문장이어야 한다.

### 9.3 AI 호출과 응답 검증 정책

- MVP 기본 모델은 `gpt-4o-mini`로 고정한다.
- 서버는 OpenAI 응답을 JSON으로 파싱한 뒤 `zod` 런타임 스키마로 검증한다.
- 검증 스키마는 `goal` 1~300자, `emotionTag` enum, `tasks` 5~10개, 각 태스크의 `title` 1~60자, `twoMinuteAction` 1~160자를 요구한다.
- 알 수 없는 필드, 빈 문자열, 허용되지 않은 감정 태그, 태스크 개수 초과/미달, 중복 태스크는 실패로 처리한다.
- 검증 실패, 파싱 실패, API 키 누락, OpenAI 호출 실패는 성공 데이터로 대체하지 않고 500 오류로 반환한다.
- 클라이언트는 서버 오류를 복구 가능한 UI 상태로 표시하되, 임의 태스크를 생성하지 않는다.

## 10. 감정 태그와 오브젝트 매핑

| 태그 | 의미 | 3D 표현 | 색상 | 움직임 | 2D 폴백 |
| --- | --- | --- | --- | --- | --- |
| `burden` | 부담스러운 일 | `DodecahedronGeometry` + 거친 material | 회색, 카키, 머드 | 느리고 무겁게 흔들림 | 무거운 원형 blob |
| `creative` | 창의적인 일 | sphere 기반 distorted blob | 파스텔, 무지개 톤 | 경쾌하게 튀어오름 | 색상 그라디언트 blob |
| `difficult` | 까다롭고 복잡한 일 | `IcosahedronGeometry` + spike/normal distortion | 레드, 퍼플 | 작게 떨림 | 뾰족한 다각형 |
| `urgent` | 긴급한 일 | cone 또는 tetrahedron cluster 기반 flame/spike | 오렌지, 레드 | 빠른 진동 | 불꽃형 CSS shape |
| `routine` | 지루하고 반복적인 일 | 납작하게 scale한 sphere/blob | 베이지, 연회색 | 거의 정지 | 낮은 대비의 납작한 blob |
| `new` | 새로 시작하는 일 | seed sphere + 간단한 sprout curve/leaf mesh | 연두, 민트 | 천천히 성장 | 새싹형 원/잎 조합 |

완료율은 별도 태그를 바꾸지 않고 같은 태그 안에서 `roughness`, `noise`, `speed`, `scale`, `opacity`를 점진 조정한다.

`blob`, `spike`, `flatBlob`, `seed`는 Three.js/drei의 완성 컴포넌트 이름이 아니라 구현 레시피다. MVP에서는 공통 primitive를 변형해 여섯 태그의 느낌을 구분하고, 정교한 절차적 모델링은 후속 개선으로 둔다.

### 10.1 3D/2D 선택 규칙

아래 조건 중 하나라도 만족하면 2D 폴백을 우선한다.

- WebGL context 생성 실패
- `prefers-reduced-motion: reduce`
- viewport 폭 768px 미만
- `navigator.deviceMemory`가 존재하고 4GB 미만

`navigator.deviceMemory`는 일부 브라우저에서만 제공되므로 단독 기준으로 사용하지 않는다.

## 11. 상태 관리 요구사항

### 11.1 Zustand 저장 대상

- `session`
- `currentIndex`
- 현재 태스크 상태 변경 액션
- 기분 이모지 저장 액션
- 세션 생성/복구/폐기 액션

### 11.2 React 로컬 상태 대상

- 입력값
- API 호출 중 로딩 상태
- 일시적 오류 메시지
- 파티클/전환 애니메이션 재생 여부
- 모달/프롬프트 표시 여부

### 11.3 LocalStorage 키

권장 키: `ddak.task-session.v1`

저장 시점:

- 분석 성공 후 세션 생성
- 시작 버튼 클릭
- 완료 버튼 클릭
- 기분 이모지 선택
- 모든 태스크 완료
- 세션 폐기

복구 조건:

- 세션이 존재하고 완료되지 않은 태스크가 하나 이상 있으면 복귀 프롬프트를 표시한다.
- 세션에 `completedAt`이 있으면 복귀 프롬프트 대신 완료 축하 상태 또는 새 목표 시작 CTA를 표시한다.
- 저장 데이터가 현재 스키마 버전과 호환된다.
- 저장 데이터 파싱 또는 스키마 검증에 실패하면 자동 삭제하지 않고, "저장된 진행 상태를 불러올 수 없어요" 안내와 세션 초기화 CTA를 보여준다.
- 사용자가 초기화를 선택한 경우에만 `ddak.task-session.v1`을 제거하고 새 목표 입력 화면으로 이동한다.

### 11.4 LocalStorage 저장 스키마

```ts
type PersistedTaskSession = {
  schemaVersion: 1;
  session: TaskSession;
};
```

LocalStorage 키에 이미 버전(`v1`)이 포함되어 있어도 저장 payload에 `schemaVersion`을 함께 둔다. 구현은 저장소 복구 시 `schemaVersion === 1`과 `TaskSession` 구조를 모두 확인한다.

## 13. 구현 단계

- 앱 이름과 기본 문구가 딱(Ddak) 기준으로 바뀐다.

### Phase 2. AI 분석 API

- `POST /api/analyze` 추가
- OpenAI 서버 호출 구현
- JSON 응답 검증
- rate limit 적용
- 입력 오류와 서버 오류 분리

- 유효한 목표는 5~10개 태스크와 단일 감정 태그를 반환한다.
- API 키가 없거나 AI 응답이 잘못되면 명시적 오류를 반환한다.
- 클라이언트에 API 키가 노출되지 않는다.

### Phase 3. 상태와 저장소

- Zustand store 추가
- LocalStorage persistence 추가
- 세션 복구/폐기 액션 구현
- 완료와 이모지 저장 액션 구현
- 저장 데이터 파싱/스키마 실패 시 초기화 프롬프트 구현

수용 기준:

- 새로고침 후 미완료 세션을 복구할 수 있다.
- 완료된 세션은 복귀 프롬프트를 띄우지 않는다.
- 새 목표 시작 시 기존 세션을 명시적으로 폐기할 수 있다.

### Phase 4. 핵심 UX

- 목표 입력 화면 구현
- 분석 로딩 화면 구현
- 현재 태스크 카드 구현
- 남은 태스크 흐림 표시
- 시작/완료 플로우 연결
- 이모지 선택/건너뛰기 플로우 연결
- 전체 완료 화면 구현

수용 기준:

## 17. 구현자가 지켜야 할 주의점

- AI 응답 실패를 임의 성공 데이터로 대체하지 않는다.
- LocalStorage 파싱 실패는 사용자에게 세션 초기화 선택지를 제공한다.
- 3D 구현이 지연되어도 목표 입력, AI 분석, 태스크 진행은 독립적으로 동작해야 한다.
- 이모지 건너뛰기는 `neutral`로 자동 치환하지 않는다. 사용자가 선택하지 않은 감정을 저장하지 않는다.
- 감정 태그 라벨과 오브젝트 프로필은 중복 정의하지 않는다.
- 현재 설치된 Next.js 16 기준 문서를 확인한 뒤 Route Handler, dynamic import, 클라이언트 컴포넌트를 구현한다.
- 접근성, 저모션, 모바일 폴백은 후순위 장식이 아니라 MVP 수용 기준에 포함한다.
